Help & Troubleshooting

Common setup issues and what the error messages on each tab actually mean. Expand a section below.

WSJT-X GUI bridge errors (Remote tab: Enable Tx / Tune / Monitor / Decode / Stop / band+mode buttons)

Symptom: red text under the button row, e.g. "osascript is not allowed assistive access. (-1719)", or "Can't get checkbox... (-1728)" during startup.

These controls have no WSJT-X UDP equivalent, so this app drives WSJT-X's own window directly via macOS Accessibility scripting -- System Events needs permission to read/click WSJT-X's UI, which macOS grants per-process and can lose track of after this app's Python process restarts (it isn't a signed app bundle, so a fresh process instance sometimes isn't recognized as already trusted).

Fix -- System Settings → Privacy & Security → Accessibility, add both of these (use the + button, then Cmd+Shift+G in the file picker to paste the path) and make sure both are toggled on:

  • /opt/local/Library/Frameworks/Python.framework/Versions/3.12/Resources/Python.app
  • /usr/bin/osascript

No app restart needed -- the permission takes effect on the already-running process as soon as you toggle it on. If it was working and suddenly stops after you (or Claude) edit a .py file -- the dev server auto-restarts on save, and that's exactly the kind of restart that can drop this permission -- re-check these two entries first before assuming something else broke.

Also needs pyobjc-framework-Quartz installed (in requirements.txt) for Enable Tx/Tune specifically -- those arm via a real synthetic click, not just an accessibility property read.

Radio tab: rigctld not connected

Symptom: red dot / "Not connected" at the top of the Radio tab, or the rigctld process panel shows "rigctld binary not found".

  1. Install Hamlib if it isn't already: port install hamlib or brew install hamlib.
  2. In the "rigctld process" panel, set a rig model and serial device (the device dropdown lists /dev/cu.* ports actually present).
  3. Click Start. If it fails, expand "rigctld log" underneath for the actual CAT-level error -- usually a wrong serial device, wrong baud rate, or the radio's USB/CAT port not enabled.

If rigctld is already running from somewhere else (not started by this app), the panel shows "Running (started outside this app)" and Stop/Restart are unavailable -- that's expected, not an error.

RigDial tab: dial not detected / hardware control disabled

Symptom: red dot, "Not connected -- plug in the dial", or "hid package/native library not available".

This needs two separate things, both listed in requirements.txt:

  • The hid Python package (pip install hid).
  • A native hidapi library -- pip does not provide this, it has to be installed separately: port install hidapi or brew install hidapi.

If the status specifically says the package/library is unavailable, that's a software install problem (fix via the commands above) -- if it just says "not connected", the software side is fine and it's genuinely not seeing the ShuttleXpress (check it's plugged in, try a different USB port).

The frequency preset list works from the browser regardless -- "Set Now" doesn't need the dial plugged in at all, only the button/jog/shuttle mapping does.

Live Monitor / Remote / Waterfall / QSL Helper: no decodes, "not connected"

Symptom: decode tables stay empty, connection status never goes green.

These all depend on WSJT-X broadcasting its UDP status/decode messages. In WSJT-X: Settings → Reporting, enable "Accept UDP requests" and check the UDP Server host/port matches what this app listens on (127.0.0.1:2237 by default, configured in app.py) -- and if you're using multicast, that the multicast group (224.0.0.1 by default) matches too.

WSJT-X must actually be running and actively decoding (or at least connected) for any of this to show data -- a closed WSJT-X, or one that hasn't decoded anything yet this session, will show as quiet/empty rather than erroring.

Remote tab: queued QSOs won't send to MacLoggerDX

Symptom: entries in the QSO queue stay stuck in "pending" or show an error after clicking Send.

QSOs are pushed into MacLoggerDX via its own importADIF AppleScript command, which needs Automation permission -- a different permission category from Accessibility (above), specific to one app controlling another. Check System Settings → Privacy & Security → Automation, find Python, and make sure MacLoggerDX is allowed underneath it. MacLoggerDX also needs to actually be running.

Why does the app keep reconnecting / briefly go offline while being worked on?

The app runs in debug/auto-reload mode -- saving a change to any .py file restarts the whole Flask process automatically (you'll see every tab's status briefly drop to "connecting..." before reconnecting on its own within a second or two). This is expected during development, not a fault. As noted above, this kind of restart is also the most common trigger for the Accessibility permission dropping out, so check that first if the WSJT-X GUI bridge stops working right after an edit.
